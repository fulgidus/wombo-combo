## The actual bugs — precise diagnosis

**Bug 1 — The concurrency display shows 3 when you enter the quest.**

On `bun dev`, the daemon starts fresh. `state.load()` finds no `daemon-state.json` (or finds one from a prior run with `maxConcurrent: 3`). **Bug 2 — You set concurrency to 0 (infinite), press A, tasks still launch 3 by 3.**

This is the real scheduling bug, and the `cmd:set-concurrency` path is correctly implemented. So why doesn't it work? The question is: **does `state.getMaxConcurrent()` actually return `0` by the time the scheduler tick runs?**

Two candidates:
- `cmd:set-concurrency` message arrives, daemon calls `state.setMaxConcurrent(0)`, `scheduleSave()` debounces 100ms — but in that 100ms window, does the tick fire and read the old value? Unlikely but possible.
- More likely: **the daemon is being restarted** (not just the TUI reconnecting) each time you `bun dev`. If `daemon-state.json` exists from the last run and has `maxConcurrent: 3` baked in, `state.load()` restores `3`. You set it to `0`, but then you `bun dev` again for the quest test and the whole cycle repeats.

**Bug 3 — The root of your frustration.** `config.defaults.maxConcurrent` should be a *bootstrap value*, not a *reset value*. ## On Elixir/BEAM architecture as an evolution

Genuinely interesting question. Here's an honest evaluation:

**What BEAM gives you that you don't have:**
- **Supervisors with restart strategies** — if the Scheduler crashes mid-tick, a supervisor restarts it with clean state. Right now a JS exception in the tick loop can silently kill the whole scheduler.
- **Process isolation** — each agent could be a supervised process; a crash in one doesn't affect others. Currently they're just tracked entries in a Map.
- **Location transparency** — messages between processes work the same whether local or distributed. Not relevant now, but if you ever wanted multi-machine orchestration it's free.
- **"Let it crash" philosophy** — instead of defensive `try/catch` everywhere, you write happy-path code and let supervisors handle failures. Your codebase has a lot of `catch { /* best-effort */ }` which is the opposite.

**What BEAM does NOT give you for this specific bug:**
- The `maxConcurrent` bug is a state ownership problem. BEAM would not fix it — you'd just have a `GenServer` holding the value instead of a class, and the reset-on-restart bug would still exist unless you explicitly design around it.

**TypeScript actor libraries:**

| Library | Status | Verdict |
|---|---|---|
| `nact` | Largely abandoned, last release 2020 | Skip |
| `xstate` v5 | Active, excellent, widely used | Relevant but it's a state machine library, not full actor model |
| `@temporalio/worker` (Temporal) | Active, production-grade | Overkill for this, but genuinely BEAM-like for long-running workflows |
| `@dapr/dapr-client` | Active | Distributed actors, way too heavy |
| Custom using `worker_threads` + `MessageChannel` | Always available in Node/Bun | Closest to real BEAM isolation without a framework |

**My honest take:** The architecture you have — a single daemon process with in-memory state — is not the problem. The problem is that **`DaemonState` has no ownership rules**: any part of the code can call `setMaxConcurrent()` at any time, and multiple callers (Scheduler.start, cmd:set-concurrency, handleStart reconstruction) fight over the same field. 

BEAM would give you better fault isolation (worth having eventually), but the immediate fix is simpler: **`maxConcurrent` should be set exactly once on first daemon boot, and after that only changed by explicit user commands (`cmd:set-concurrency`).** The Scheduler should never write to it. **What would actually help from BEAM thinking:**
- Treat `DaemonState` as a `GenServer`-style single-owner — only one actor writes to each field, everyone else reads

- Use `worker_threads` for agent processes instead of `Bun.spawn` subprocesses — true isolation, shared memory possible, cleaner lifecycle

## What we did

### Root cause confirmed

The "3-at-a-time" bug was caused by `Scheduler.start()` unconditionally overwriting `state.maxConcurrent` with `config.defaults.maxConcurrent` (3) on every daemon boot. This happened because:

1. `state.load()` restores the user's persisted `maxConcurrent = 0` (infinite)
2. `new Scheduler(...)` creates a fresh instance with `concurrencyPinned = false`
3. `scheduler.start()` sees `concurrencyPinned = false` → writes `3` over the restored `0`
4. User's persisted "infinite" setting is silently lost before the TUI even connects

The `concurrencyPinned` flag was designed to prevent within-session overwrites but didn't protect against the boot-time reset.

### Fix applied (`fix-concurrency-boot-reset`)

**`src/daemon/state.ts`** — Added `stateLoaded: boolean` field; set to `true` in `load()` when a valid `daemon-state.json` is found and parsed.

**`src/daemon/daemon.ts`** — In `Daemon.start()`, after `state.load()`, pre-set `scheduler.concurrencyPinned = true` when `state.stateLoaded` is true. This means: if persisted state was loaded, the scheduler's `start()` skips the config-default write entirely.

**`tests/daemon-scheduler.test.ts`** — Added 2 new tests: one verifying that `concurrencyPinned = true` prevents the config default from overwriting a persisted value, one verifying the config default still applies on fresh boot.

- `bun run typecheck` — clean
- `bun test` — 2212 pass, 12 pre-existing TUI snapshot failures (no regressions)
